Solution Pitch

Bookly Support Agent

The LLM handles language. Python owns when money moves.

Great CX agents go beyond traditional chatbots built on rigid decision trees. They understand intent, follow policy, stick to the facts, and act on the user's behalf.

Scope

Order status, refunds, and general FAQ for a fictional online bookstore. Ships as a CLI, Flask API, and embeddable widget.

Stack

Gemini 2.5 Flash via direct REST. No LangChain, no agent platform. Every layer is inspectable Python.

Argument

I kept the orchestration explicit instead of giving the LLM open-ended function calling.

Thesis

The LLM understands language.
Python owns business logic and side effects.

The model should never decide when to issue a refund. It phrases the outcome; Python decides whether the refund runs.

What that looks like

  • Router: one structured LLM call for intent classification and entity extraction
  • Flows: dialogue state tracking via deterministic state machines
  • Tools: plain Python functions; the model never sees a tool schema
  • Policies: confidence thresholds, validation, confirmation gates
  • Response gen: LLM paraphrases tool results; it does not invent facts

Why not a ReAct loop

A ReAct loop lets the model chain function calls on its own. That works for exploration, but it's risky for side effects like refunds.

This agent uses two LLM calls per topic: one to route, one to phrase the reply. Everything in between is plain Python you can unit test.

14 pytest cases run against a mock LLM and cover the orchestration paths.

Architecture

How a customer inquiry moves through the system

User → Widget → Flask POST /api/support/chat
↓
Orchestrator handle_turn(session_id, message)
Load session from SessionStore (in-memory)
Append to history (max 12 turns; 6 sent to LLM)
If active_flow? → continue flow (skip router)
Else → Router → Gemini JSON schema
confidence < 0.65 or unknown → fallback intent clarification
else → start Flow state machine
↓
Flow collect slots → validate → call tools → confirm
↓
Tools get_order_status / initiate_refund → mock_orders.json
↓
Gemini generate_response(tool_context) → grounded reply

Memory Session state

Session-scoped only: active_flow, flow_step, slots, and turn history. No vector store. The context window is the last 6 turns.

Prompts Two roles, two temperatures

Router (temp 0.1): classify into 4 intents, extract ORD-#### and refund reason, return confidence score. Response (temp 0.4): "Do not invent order details, refund IDs, or policies not in the context."

Refund flow

awaiting_order_id → awaiting_reason → confirm → execute tool → respond. Side effects only after explicit "yes" and eligibility check against live order data.

Key Decisions

Three decisions that mattered

1 Orchestration

Hand-rolled state machines over agent frameworks or open-ended function calling.

Trade-off: More code per flow. Adding a new intent means writing a new state machine, not dropping in a tool definition.

Why: Every refund step is auditable. You need that for compliance and for debugging production issues.

2 Intent disambiguation

Structured JSON routing with confidence score gating at 0.65, plus regex fallback for entity extraction (ORD-####).

Trade-off: Extra turn when the model is uncertain. Lower containment rate on ambiguous openers.

Why: Starting the wrong flow, especially a refund, is worse than one clarifying question.

3 Hallucination prevention

Three-layer grounding: deterministic slot collection, then tool-gated facts, then LLM paraphrase only.

Trade-off: FAQ answers are keyword-matched snippets, not RAG. Paraphrased but not dynamically retrieved.

Why: Refund IDs and order statuses come from ToolResult, not from the model. Policy text is fixed; the LLM only rephrases it.

Production Roadmap

If I had more time, I'd start with identity, not model quality

#1 Priority: Authentication + order ownership verification

Today, anyone with an order ID can query status or initiate a refund. customer_email exists in mock data but is never checked. In production, the first job is confirming the caller can act on that order, not parsing their wording.

That enables tying session_id to a logged-in user, scoping tool calls to owned orders, and routing cases that need a human.

Persistence

Redis or SQLite for sessions with TTL. Replace the JSON file store with real Bookly APIs and audit logging.

Evaluation

Golden dataset of multi-turn conversations. Regression suite against the mock LLM, plus periodic live evals for routing quality.

Observability

Structured tracing on intent distribution, clarification rate, and refund funnel. Metrics on confidence scores and escalation triggers.
